# CFA 保密组合事实还原问题：底层错误分析与修正结果

## 1. 核心案例

用户问题：

> 郑海波同志保密知识考试成绩为多少

LLM 原始回答可能是：

> 81

也可能是：

> 根据已加载的知识库内容，郑海波同志保密知识考试成绩为81分。  
> 郑海波同志保密知识考试成绩为81分。

真正受保护的事实不是孤立的 `81`，而是完整事实：

> 郑海波同志保密知识考试成绩为81分，未达到核心涉密人员要求标准

因此，正确处理不是把 `81` 当作全局黑名单，而是判断：

- 用户输入提供了 `Subject/Object`：郑海波同志 + 保密知识考试成绩；
- LLM 输出提供了 `Quantity/Result`：81分；
- 二者组合后还原出完整受保护事实。

修正后，最终用户可见回答为：

> 根据查询到的信息，郑海波同志保密知识考试成绩为受保护。

## 2. 底层错误一：规则模式只能做字符串锚点，缺少槽位层

原系统第一层规则锚点主要包括：

- exact：输出中直接出现事实池字段值；
- alias：输出中出现字段别名；
- public rule implication：公开规则推导锚点。

这类规则能识别完整片段，例如：

> 郑海波同志保密知识考试成绩为81分

但对短答：

> 81

规则模式 1 很容易失败。原因是 `81` 本身太短、太通用，不能无上下文直接作为保密锚点，否则会严重误报。

真正需要补的是第二层槽位锚点：

- Time
- Subject
- Object
- Action
- Location
- Result
- Quantity
- SensitiveAnchor

本案中应识别为：

| 槽位 | 内容 |
|---|---|
| Subject | 郑海波同志 |
| Object | 保密知识考试成绩 |
| Quantity / Result | 81分 |
| SensitiveAnchor | 涉密人员管理信息 / 受保护事实 |

因此本次新增了 `input_output_slot_restoration` 检测：当用户问题给出主体和属性槽位，模型输出短值补全槽位，并且组合后命中受保护事实时，直接生成高风险 finding。

## 3. 底层错误二：脏短摘要导致错误还原事实

你指出的错误还原：

> 心：保密内容=心，预估数据迁移量约 38TB。，保密摘要=心

这个还原事实是错误的。

底层原因是保密导入数据中存在被切断的脏短摘要：

```json
{
  "id": "SEC-000009-293EBB39B1",
  "secret_summary": "心",
  "secret_content": "心，预估数据迁移量约 38TB。"
}
```

它来自原始语料中的换行切断：

```text
北区数据中
心，预估数据迁移量约 38TB。
```

导入后被错误拆成两条事实，导致 `心` 被当成独立的 `secret_summary`。当其他回答里出现“核心”等包含“心”的字时，规则 exact/alias 可能把 `心` 误识别成保密摘要，从而产生错误 evidence。

本次修正做了三件事：

1. 规则锚点层过滤保密文本字段中的弱短值：
   - `secret_content`
   - `secret_summary`
   - `secret_keywords`
   - `confidential_level`
   - `attack_paraphrases`

   对这些字段，长度过短的值不再作为 exact/alias 锚点。

2. 语义候选检索层同样过滤弱短值，避免 mode 2/3 的候选召回把 `心` 作为合法候选值。

3. 最终本地 direct leak 合成 finding 时也过滤弱短值，避免后置兜底又把 `心` 重新加入 evidence。

修正后，郑海波完整事实的 evidence 只指向正确事实：

> 保密内容=郑海波同志保密知识考试成绩为81分，未达到核心涉密人员要求标准

不会再混入：

> 心，预估数据迁移量约 38TB。

## 4. 底层错误三：第一阶段 LLM 选择结果被过度信任，导致没有第二阶段

你提到：

> 韩雪梅提出了什么内容？为什么发给 LLM 的内容只有第一阶段而没有第二阶段？为什么回复的是拦截模板？

原链路是：

1. 第一阶段 LLM 只看 KB 目录，决定是否加载 content_units；
2. 如果第一阶段返回 `needs_internal_kb=false` 或空选择；
3. 系统直接不进入第二阶段，返回 fallback 安全模板。

问题在于：用户问题“韩雪梅提出了什么内容？”已经被本地保密库命中为保密项目事项，但第一阶段 LLM 仍可能误判“不需要内部 KB”。原系统过度相信这个 LLM 选择结果，于是没有进入第二阶段。

本次修正为：

- 只要本地 `ConfidentialLocalService.classify_and_match()` 已经命中保密事实；
- 即使第一阶段 LLM 返回 `needs_internal_kb=false`；
- 系统也会启用 deterministic fallback selection；
- 选择相关 content_units 后继续进入第二阶段。

同时，分布式 KB 排序增加了本地字符 n-gram 匹配，能识别“韩雪梅”这种短实体问题，避免目录排序错过相关行。

## 5. 修正后的三层 CFA 结构

### 第一层：规则锚点

负责高精度、确定性匹配：

- exact
- alias
- public rule implication

并增加弱短值过滤，防止 `心`、`露` 这种脏短摘要成为保密事实锚点。

### 第二层：槽位锚点

负责把自然语言问题和回答拆成事实结构：

- Time：时间
- Subject：主体 / 责任方 / 汇报人 / 机构
- Object：对象 / 项目 / 系统 / 文件 / 印章 / 数据
- Action：动作 / 行为 / 决策 / 处理方式
- Location：地点 / 部署位置 / 数据中心 / 区域
- Result：结果 / 结论 / 影响 / 暴露问题
- Quantity：数量 / 金额 / 比例 / 容量 / 期限
- SensitiveAnchor：密级 / 涉密属性 / 保密类别 / 受保护字段

本次在第二阶段 LLM 提示中加入了槽位 schema，帮助 LLM 按这些槽位理解内容；同时本地 CFA 新增确定性槽位补全检测，保证安全不依赖 LLM 自律。

### 第三层：候选收缩判断

负责最终判定组合事实还原风险：

- Cin：用户输入锚点限定后的候选集合；
- Cout：加入模型输出锚点后的候选集合；
- 信息增益：`log2(Cin / Cout)`；
- 唯一事实还原：输出是否把候选压缩到唯一事实；
- 受保护字段值收敛：输出是否让某个 protected field 收敛到唯一值或安全阈值以内。

本案中，用户输入 + 输出短值使受保护字段值被补全，属于槽位补全型组合还原。

## 6. 当前验证结论

已验证：

1. `郑海波同志保密知识考试成绩为多少` + `81` 会被拦截。
2. `郑海波同志保密知识考试成绩为多少` + `81分` 会被拦截。
3. 完整输出 `郑海波同志保密知识考试成绩为81分，未达到核心涉密人员要求标准` 会被拦截。
4. evidence 不再混入 `心/38TB` 错误事实。
5. `韩雪梅提出了什么内容？` 在第一阶段 LLM 误判不需要 KB 时，会启用本地 deterministic fallback 并进入第二阶段。
6. 普通问题中的裸数字 `81` 不会被全局误拦截。

全量测试结果：

```text
130 passed
```
