# CFA-Score 输出侧组合事实还原风险检测与防泄密网关系统——专利核心知识点提炼

生成日期：2026年7月9日

## 1. 一句话概括

本系统是一个部署在大模型回答输出侧的防泄密网关。它不只检查模型回答中是否直接出现敏感字段，还会把“用户输入中的线索”和“模型输出中的线索”合并分析，判断这些线索组合后是否会把内部受限事实池中的候选记录压缩到唯一或少量记录，从而间接还原低权限用户不应获得的受限事实；一旦检测到风险，系统会对模型回答进行脱敏、安全泛化、必要时进行 LLM 改写并再次检测，最终仅返回安全回答。

## 2. CFA 是什么

在本项目中，CFA 指 Combination Fact Attribution / Combination Fact Restoration 的工程化检测概念，中文可表述为“组合事实归因”或更贴近项目目的的“组合事实还原”。它不是统计学中的 Confirmatory Factor Analysis（验证性因子分析）。

CFA 的核心含义是：单个事实线索本身可能不是秘密，例如“上海”“半导体”“ICU”“项目贷款”“某个版本号”等，但当用户输入和模型输出中的多个线索组合后，可能会在内部事实池中逐步过滤候选对象，最终唯一定位某条记录，进而推知该记录中的受限字段。CFA-Score 就是对这种组合还原过程进行检测、量化评分和安全处置的机制。

可以把 CFA 理解为一个候选集压缩过程：

用户输入锚点 + 模型输出锚点 → 对内部事实池逐步过滤 → 候选集合从 N 条变为 K 条 → 若 K 小于等于安全阈值且涉及受限字段，则判定存在组合事实还原风险。

## 2.1 CFA 的核心计算思想与核心公式

CFA 的核心观察点不是“模型回答中是否出现了某个敏感词”，而是观察模型输出在用户输入已经限定的候选范围之上，是否进一步提供了能够压缩内部受限事实池的新增事实线索。换言之，CFA 关注的是“输入已有信息 + 输出新增信息”组合后是否能够还原受限记录或受限字段值。

### 2.1.1 基本符号定义

设内部受限事实池为 F，其中每一条记录 r 表示一个受保护对象，例如患者记录、贷款记录、运维资产记录或会议记录。系统从用户输入中抽取的事实锚点集合记为 Ain，从模型输出中抽取的事实锚点集合记为 Aout。

- F：内部受限事实池。
- r：事实池中的一条候选记录。
- Ain：用户输入锚点集合，表示用户在问题中已经给出的线索。
- Aout：模型输出锚点集合，表示模型回答中新给出的线索。
- Acontrib：输出贡献锚点集合，即真正导致候选集合继续收缩的模型输出锚点。
- Cin：输入侧候选集合，仅由用户输入锚点过滤得到。
- Cout：输出组合后的最终候选集合，由模型输出锚点在 Cin 基础上继续过滤得到。
- k：唯一性阈值，通常取 1。
- protected_fields：受限字段集合，例如诊断结果、用药方案、贷款金额、信用评级、组件版本、风险状态等。

### 2.1.2 输入侧候选集合 Cin

系统首先只使用用户输入锚点 Ain 对事实池 F 进行过滤，得到输入侧候选集合 Cin：

Cin = { r ∈ F | r 满足 Ain 中全部锚点约束 }

该步骤用于刻画用户在提问时已经限定出的候选范围。其专利意义在于：系统不会把用户已经提供的信息错误归因于模型泄露，而是先计算用户输入本身能够缩小到什么程度。

### 2.1.3 输出组合候选集合 Cout

随后，系统在 Cin 的基础上，使用模型输出锚点 Aout 继续过滤，得到最终候选集合 Cout：

Cout = { r ∈ Cin | r 满足 Aout 中全部锚点约束 }

如果 |Cout| 明显小于 |Cin|，说明模型输出相对于用户输入新增了具有定位作用的事实线索。CFA 的关键即在于评估这种“输出新增压缩贡献”。

### 2.1.4 输出贡献锚点 Acontrib

并非所有模型输出锚点都会造成泄露风险。系统仅将真正使候选集合数量减少的输出锚点记为贡献锚点。对按顺序处理的某个输出锚点 ai，若其过滤前候选集合为 Cbefore，过滤后候选集合为 Cafter，则满足以下条件时，ai 被记入 Acontrib：

0 < |Cafter| < |Cbefore|

其中：

Cafter = { r ∈ Cbefore | r 满足 ai 的锚点约束 }

通过 Acontrib，系统能够区分“模型只是重复用户输入中的信息”与“模型新增了足以收缩候选集合的信息”。

### 2.1.5 信息增益公式

模型输出带来的候选压缩信息量用信息增益 IG 表示：

IG = log2(|Cin| / |Cout|)

当 |Cin| > |Cout| > 0 时，IG 表示模型输出新增线索使候选集合压缩的 bit 数。例如，若 Cin 有 16 条记录，Cout 只剩 1 条记录，则：

IG = log2(16 / 1) = 4 bits

该公式是 CFA 的核心计算公式之一，用于把“候选集合被压缩了多少”转化为可量化的风险因子。

### 2.1.6 唯一性风险公式

系统进一步计算唯一性得分 U，用于描述最终候选集合是否已经唯一或接近唯一：

U = 1，若 |Cout| ≤ k

U = 1 / |Cout|，若 |Cout| > k

其中 k 为唯一性阈值。当 |Cout| ≤ k 时，表示用户输入与模型输出组合后已经能够唯一或近似唯一定位受限记录，风险最高；当 Cout 中仍存在较多候选记录时，唯一性风险随候选数量增大而下降。

### 2.1.7 受限字段值熵与熵下降公式

除唯一定位整条记录外，CFA 还关注另一类风险：即使 Cout 中仍有多条候选记录，但某个受限字段的可能取值已经被模型输出压缩到唯一或少量值。对任一受限字段 f，在候选集合 C 上定义字段值熵 Hf(C)：

Hf(C) = - Σ p(v) log2 p(v)

其中 v 表示字段 f 的一个可能取值，p(v) 表示候选集合 C 中字段 f 取值为 v 的记录比例。

模型输出导致的字段值熵下降定义为：

ΔHf = Hf(Cin) - Hf(Cout)

若满足：

|Vf(Cout)| ≤ kv，且 ΔHf ≥ τH

则认为模型输出虽然未必唯一定位整条记录，但已经使受限字段 f 的可能取值集合收敛到安全阈值以内，构成间接受限字段值还原风险。其中 Vf(Cout) 表示 Cout 中字段 f 的不同取值集合，kv 表示受限字段值收敛阈值，τH 表示最小熵下降阈值。

### 2.1.8 输出锚点贡献度公式

为了避免低置信度语义锚点或模糊匹配造成误判，系统对输出贡献锚点计算平均可信贡献度 OC：

OC = avg(conf(ai) × wtype(ai))，ai ∈ Acontrib

其中 conf(ai) 为锚点 ai 的置信度，wtype(ai) 为匹配类型权重。示例性地，精确匹配 exact 的权重可高于别名匹配 alias，别名匹配又高于语义匹配 semantic、部分匹配 partial 或模糊匹配 ambiguous。

### 2.1.9 CFA-Score 总评分公式

在上述计算基础上，系统将风险触发类型、唯一性、信息增益和输出锚点贡献度组合为 CFA-Score：

CFA-Score = min(100, Btype + αU · U + αIG · G + αOC · OC)

其中：

G = min(1, IG / IGmax)

- Btype：风险触发类型基础分。
- U：唯一性得分。
- G：归一化信息增益得分。
- OC：输出贡献锚点的平均可信贡献度。
- αU、αIG、αOC：对应权重系数。
- IGmax：信息增益归一化上限。

作为一种实施例，可设置：

CFA-Score = min(100, Btype + 25U + 15G + 10OC)

其中 Btype 可按风险类型取值：

- 直接受限字段泄露：Btype = 55。
- 间接受限记录还原：Btype = 50。
- 间接受限字段值还原：Btype = 40。

风险等级可按分值映射为：85 分以上为 CRITICAL，70 分以上为 HIGH，45 分以上为 MEDIUM，低于 45 为 LOW。

### 2.1.10 三类核心风险触发条件

基于上述公式，CFA 的核心风险触发条件可归纳为三类：

1. 直接受限字段泄露：模型输出锚点 Aout 中直接包含 protected_fields 中的字段值。
2. 间接受限记录还原：Acontrib 非空，且 |Cout| ≤ k，且 IG 达到预设最小信息增益阈值。
3. 间接受限字段值还原：模型输出未必唯一定位记录，但使某个受限字段 f 满足 |Vf(Cout)| ≤ kv 且 ΔHf ≥ τH。

因此，CFA-Score 的本质是：在双源锚点隔离的基础上，通过 Cin/Cout 候选集合压缩、信息增益、唯一性和受限字段熵下降，量化模型输出对组合事实还原风险的新增贡献。

## 3. 系统要解决的技术问题

现有输出安全检测通常关注模型是否直接说出了敏感词、身份证号、病历号、金额、系统名称等显式敏感内容，但在企业内部知识库、医疗系统、金融信贷系统、运维资产系统等场景中，真正危险的泄密经常不是单个字段的直接泄露，而是多段普通信息组合后的间接还原。

本系统解决的核心技术问题包括：

1. 如何识别用户输入和模型输出中分散出现的事实线索，并将其映射到内部事实池字段值。
2. 如何判断这些线索组合后是否足以唯一定位内部受限事实记录。
3. 如何量化模型输出对候选集合收缩的贡献，而不是把用户输入中已有的信息也算作模型泄露。
4. 如何识别直接泄露、间接记录还原、受限字段值收敛等多种风险形态。
5. 如何在检测到风险后仅改写模型输出中的危险部分，并通过二次检测验证改写结果是否仍有残留风险。

## 4. 系统在干什么

系统工作在“用户请求—大模型回答—返回用户”之间，相当于一个输出侧安全网关。完整链路如下：

1. 接收用户输入 user_input。
2. 获取或生成模型原始回答 model_output/raw_answer。
3. 加载当前业务场景的内部受限事实池 fact_pool/assets 和字段策略 policy。
4. 从用户输入和模型输出中分别抽取事实锚点 anchor。
5. 使用用户输入锚点先过滤事实池，得到输入侧候选集合 Cin。
6. 再使用模型输出锚点继续过滤 Cin，得到最终候选集合 Cout。
7. 判断模型输出是否新增了足以压缩候选集合、唯一定位记录或还原受限字段的线索。
8. 计算 CFA-Score 风险分和风险等级。
9. 生成还原链、关键锚点、被还原事实等可解释报告。
10. 对模型输出执行安全改写，必要时对改写结果再次运行 CFA 检测。
11. 对外仅返回安全回答 answer，同时可在内部审计报告中保留风险发现和还原链。

## 5. 输入是什么

### 5.1 业务请求输入

- user_input：用户输入，即用户向模型提出的问题或描述。该文本可能包含范围、类别、时间、位置、对象特征等线索。
- model_output/raw_answer：模型原始回答。可由调用方直接提供，也可由系统调用 DeepSeek 等大模型根据用户输入和事实池生成。

### 5.2 数据与策略输入

- fact_pool/assets：内部受限事实池，每条记录代表一个受保护对象，例如患者记录、贷款申请记录、运维资产记录、会议记录等。
- policy：字段策略配置，至少包括：
  - protected_fields：受限字段，例如诊断结果、用药方案、医保类型、贷款金额、利率、信用评级、抵押物、组件版本、风险状态等。
  - identifier_fields：强标识字段，例如患者姓名、企业名称、系统名称、会议主题等。
  - quasi_identifier_fields：准标识字段，例如科室、病房类型、行业、贷款类型、分行、部署环境、会议室、时间段等。
  - field_order：字段过滤顺序。
  - field_aliases：别名映射，用于把自然语言表达映射为标准字段值。
  - public_rules：公开知识推断规则，用于根据公开或上下文信息推导出附加锚点。
  - field_weights：字段权重，用于风险评分。
  - uniqueness_k：唯一还原阈值，通常为 1。
  - safe_replacements/safe_hint：安全替换词和授权查询提示。
- public_knowledge：可选公开知识规则，用于模拟公开情报或公共上下文。
- semantic_aliases：可选语义别名配置，用于 LLM 语义抽取前的候选召回。

### 5.3 控制输入

- scenario：业务场景，例如 healthcare、finance、aerospace、meetings 或 auto。
- mode：抽取模式，包括 rule_only 和 rule_plus_llm。
- secondary_check：是否启用 LLM 安全改写后的二次 CFA 检测。
- env/API 配置：用于调用外部大模型生成回答或执行语义抽取、改写。

## 6. 输出是什么

### 6.1 风险检测输出

系统内部输出 AnalysisResult，主要包括：

- anchors：从用户输入和模型输出中抽取的锚点列表。
- findings：风险发现列表。
- risk_level：风险等级，可为 NONE、LOW、MEDIUM、HIGH、CRITICAL。
- score：CFA 风险分，0 到 100。
- restored_fact：被直接或间接还原出的受限事实说明。
- key_anchor_summary：导致唯一还原的关键锚点组合。
- reduction_chain：候选集合逐步压缩链，例如 10→2→1。
- x_replaced_answer：极端 X 替换版本。
- safe_answer：安全泛化后的回答。
- secondary_safe_answer：二次检测后的最终安全回答。
- secondary_findings：对改写结果再次检测后发现的残留风险。

### 6.2 网关对外输出

HTTP/API 层输出 GatewayResponse，其中最重要的字段是 answer。调用方应只使用 answer 字段作为最终返回给用户的文本。

其他字段可用于前端展示或审计，包括：

- raw_answer：原始模型回答，仅用于内部透明展示或对比，不应作为最终返回。
- risk_detected：是否检测到风险。
- safe_answer_used：最终回答使用策略，例如 raw_answer、cfa_safe_answer、secondary_safe_answer、fallback。
- findings_count：风险发现数量。
- findings_summary：风险发现摘要、关键锚点和还原链。
- intent/routed_scenario：意图分类和实际路由场景。

## 7. 中间做了什么事情：核心处理流程

### 7.1 场景加载与字段分级

系统先根据场景加载事实池和策略文件。策略把字段分成三类：受限字段、强标识字段、准标识字段。该分级使系统不仅能识别“直接说出敏感字段”的泄露，也能识别多个准标识字段组合造成的间接定位风险。

### 7.2 模型回答生成或接收

系统可以直接接收外部传入的 model_output，也可以调用 DeepSeek 等大模型生成 raw_answer。生成时可将用户输入、公开知识、事实池和字段策略作为上下文注入，以模拟企业内部 AI 助手使用内部知识回答问题的场景。

### 7.3 双源锚点抽取

系统分别处理用户输入和模型输出，给每个锚点标记 source=input 或 source=output。

规则抽取包括三类：

1. 精确字段值匹配：把事实池中字段值在文本中的出现位置抽取为锚点。
2. 别名匹配：把“重症监护室”映射为 ICU，把“流贷”映射为流动资金贷款等。
3. 公开规则推断：例如某个公开漏洞版本可推断风险状态为高危。

LLM 语义抽取模式下，系统先用语义索引和混合稀疏召回找出可能相关的候选字段值，再把少量候选交给 LLM 做单段锚点抽取，避免把完整事实池交给抽取模型。

### 7.4 语义候选召回与 LLM 锚点校验

语义索引采用别名、组成成分、部分线索、BM25、中文 n-gram 和字段提示词进行混合稀疏召回，召回结果作为 LLM 可选择的候选白名单。

LLM 抽取出的锚点必须经过校验：字段名必须存在，证据文本必须真实出现在对应文本段中，canonical_value 或 accepted_values 必须来自事实池或候选白名单，置信度必须达到阈值，protected 标记必须以 policy 为准。这样可以降低 LLM 幻觉和跨段误归因风险。

### 7.5 源隔离与锚点合并

输入段和输出段不互相去重，即同一个词同时出现在用户输入和模型输出中，会保留两个来源不同的锚点。规则锚点和 LLM 锚点只有在来源、字段、值、文本位置等均一致时才合并。合并后保留 match_type、confidence、accepted_values、extractor_sources、merged_anchor_ids 等来源信息。

### 7.6 候选集合快照

系统先用输入锚点过滤全量事实池，得到输入侧候选集合 Cin；再用输出锚点过滤 Cin，得到最终候选集合 Cout。这样可以区分“用户本来已经提供的信息”和“模型回答新增的信息”。

系统还会识别真正造成候选收缩的输出锚点，即 contributing_output_anchors，并计算信息增益：

information_gain_bits = log2(|Cin| / |Cout|)

如果模型输出没有新增可压缩候选的信息，即使文本中出现了一些普通描述，也不会被误判为高风险。

### 7.7 组合事实还原风险判定

系统支持三类风险形态：

1. 直接受限字段泄露：模型输出直接包含 protected_fields 中的字段值。
2. 间接受限记录还原：模型输出新增线索将候选记录从 Cin 压缩到 Cout，且 Cout 的数量小于等于 uniqueness_k。
3. 间接受限字段值还原：即使未唯一定位整条记录，模型输出也可能使某些受限字段的可选值集合收敛到阈值以内，并造成明显熵下降。

该设计使系统不仅能检测“说出了敏感词”，还能检测“没有直接说，但已经足以推出来”的泄露。

### 7.8 CFA-Score 风险评分

系统根据触发类型、唯一性、信息增益和输出锚点置信度计算 0 到 100 的风险分。当前核心评分思想如下：

- 直接受限字段泄露基础分较高。
- 间接受限记录还原基础分次之。
- 受限字段值收敛基础分再次之。
- Cout 越接近唯一，唯一性得分越高。
- Cin 到 Cout 的压缩越明显，信息增益得分越高。
- 输出锚点的置信度和匹配类型权重越高，输出贡献得分越高。

风险等级映射为：85 分以上 CRITICAL，70 分以上 HIGH，45 分以上 MEDIUM，低于 45 为 LOW。

### 7.9 还原链与关键锚点解释

系统会按字段顺序构造 reduction_chain，记录每个锚点过滤前后的候选数量和剩余记录 ID。例如：行业领域=半导体制造：10→2；贷款类型=项目贷款：2→1。系统还输出 key_anchor_summary，用于说明具体是哪几个用户输入/模型输出锚点造成唯一还原。

这部分对专利很重要，因为它把风险判定从黑盒分类变成了可解释的候选压缩过程。

### 7.10 安全改写与二次检测

检测到风险后，系统只改写模型输出中的危险输出锚点，不改写用户原始输入。系统可生成两种安全版本：

1. X 替换版：把危险文本替换为 X，用于极端脱敏或审计展示。
2. 安全泛化版：按场景策略把“AA+”“具体抵押物”“具体用药方案”等替换为“相关评级”“相关抵押物”“相关治疗方案”等。

在增强模式下，系统可调用 LLM 对原回答进行安全改写，然后再次运行 CFA 检测。如果二次检测仍有残留风险，则返回兜底安全回答，例如提示用户通过授权系统查询。

## 8. 核心数据结构

### 8.1 AssetFact

表示事实池中的一条受限记录。它包含 id、显示名称和若干字段值，项目通过 extra 字段机制支持不同业务场景下的任意字段。

### 8.2 FieldPolicy

表示字段策略，是系统通用化的关键。它定义哪些字段是受限字段，哪些是强标识字段，哪些是准标识字段，以及字段顺序、别名、公开规则、权重、唯一性阈值和安全替换词。

### 8.3 Anchor

表示从文本中抽取出的一个事实线索。每个 Anchor 至少包含字段名、字段标签、原文证据、规范字段值、位置、来源、是否受限、是否推断、匹配类型和置信度。

### 8.4 CandidateSnapshot

表示候选集合快照，记录输入锚点过滤后的 input_candidates、输入加输出锚点过滤后的 final_candidates、输出贡献锚点和信息增益。

### 8.5 RiskFinding

表示一次风险发现，记录目标对象、风险等级、分数、原因、被还原事实、匹配锚点、还原链、关键锚点和还原类型。

### 8.6 AnalysisResult / GatewayResponse

AnalysisResult 是内部完整检测结果；GatewayResponse 是对外 API 返回结果，最终用户应只使用其中的 answer 字段。

## 10. 核心创新链

用户输入、模型输出、事实池和字段策略
→ A1：双源锚点建模，分别抽取 input anchors 与 output anchors
→ C2：规则/别名匹配作为基础锚点抽取桥接
→ A6：可选源隔离 LLM 语义抽取与候选白名单校验
→ A2：构造 Cin/Cout 候选集合快照并识别贡献输出锚点
→ A3：执行直接泄露、间接记录还原、受限字段值熵收敛三类风险判定
→ A4：计算 CFA-Score 并生成风险等级、关键锚点和还原链
→ A5：按风险锚点进行安全泛化或 LLM 改写并二次检测
→ 输出安全回答 answer 以及内部审计报告。

## 11. 可以写入专利的系统模块划分

1. 输入接收模块：接收用户输入、模型输出、场景标识和检测模式。
2. 场景资源加载模块：加载事实池、字段策略、公开知识和语义别名。
3. 锚点抽取模块：分别从用户输入和模型输出中抽取事实锚点。
4. 语义增强抽取模块：通过候选召回、LLM 单段抽取和校验增强锚点召回。
5. 锚点合并与源隔离模块：合并多抽取器锚点并保持输入/输出来源隔离。
6. 候选集合压缩模块：构造 Cin 和 Cout，识别贡献输出锚点。
7. 组合事实还原判定模块：判断直接泄露、间接记录还原和受限字段值收敛。
8. 风险评分模块：计算 CFA-Score 和风险等级。
9. 可解释报告模块：生成还原链、关键锚点和被还原事实。
10. 安全改写模块：生成 X 替换版、安全泛化版或 LLM 改写版。
11. 二次检测与兜底模块：对改写结果再检测，残留风险时返回兜底答案。
12. 对外服务模块：通过 HTTP API、CLI 或前端工作台返回安全回答和审计摘要。

## 13. 示例化说明

以金融信贷场景为例，用户输入“上海那边半导体行业的公司，最近申请的项目贷款和流动资金怎么样了？主要想知道信用评级和抵押物”。用户输入中的“上海”“半导体”“项目贷款”“流动资金”会被抽取为输入锚点。模型输出如果进一步说出某企业名称、AA+ 信用评级、具体抵押物等，则这些输出锚点会继续过滤事实池。

系统会先用用户输入锚点将事实池从多条记录缩小到候选集合 Cin，再用模型输出锚点继续缩小为 Cout。如果 Cout 只剩一条记录，并且输出中包含信用评级、抵押物等 protected_fields，系统就判定存在组合事实还原风险，并输出导致该风险的 key anchors 和 reduction chain。随后系统把“AA+”“具体抵押物”“具体企业名”等替换为“相关评级”“相关抵押物”“某企业”等泛化表达，并追加授权查询提示。

## 14. 专利撰写建议

建议将独立权利要求聚焦在以下最小创新链上：

接收用户输入和模型输出；分别抽取带来源标记的事实锚点；根据输入锚点生成第一候选集合；根据输出锚点生成第二候选集合；基于候选集合收缩、唯一性阈值、受限字段集合和输出贡献锚点判断组合事实还原风险；根据风险结果生成安全回答。

从属权利要求可分别展开：

1. 受限字段、强标识字段和准标识字段的策略配置方式。
2. Cin/Cout 和信息增益的计算方式。
3. 受限字段值集合熵下降的判断方式。
4. CFA-Score 的评分因子和等级映射。
5. LLM 语义锚点抽取的候选白名单和源隔离校验。
6. 风险锚点驱动的安全替换、LLM 改写和二次检测兜底机制。
7. 多场景通用事实池和策略文件的配置化适配方式。

## 15. 最终结论

该项目的专利核心不在于调用某个大模型、写 HTTP 接口或做简单关键词脱敏，而在于提出了一套面向大模型输出侧的组合事实还原风险检测机制：用输入/输出双源锚点刻画事实线索，用候选集合压缩和信息增益刻画模型输出的新增泄露贡献，用唯一性和受限字段熵收敛判断间接还原风险，再用 CFA-Score 量化风险并驱动安全改写与二次检测。该机制能够覆盖传统敏感词检测难以发现的“多线索组合推断泄密”场景，适合写成软件/算法/AI 安全网关类发明专利。
